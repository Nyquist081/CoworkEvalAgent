
我们的评测实践：吸纳官方精华，补齐行业场景
skill-creator 提供了优秀的 Skill 级评测方法论，但它的设计定位是通用的 Skill 质量验证工具。在将其落地到我们的通信行业场景时，我们发现需要在多个维度上做补充和扩展。

4.1 核心设计：以评测集 Baseline 为锚点
前两章介绍的官方方法论和 skill-creator 工具链，给了我们一系列关键启发：

对照实验思想：没有 baseline 就没有评价标准——skill-creator 通过对照组（无 Skill / 另一版本 Skill）来量化 Skill 的增益或回归
量化断言驱动：将主观判断转化为可自动验证的结构化断言，让评测可重复、可回归
多角色分工：Runner 执行、Grader 评分、Analyzer 分析，各司其职，实现采集与评判的解耦
性能指标采集：对工具调用次数、Token 消耗、执行耗时等过程指标的系统化采集和聚合
但 skill-creator 的 baseline 是运行时动态生成的——每次评测临时跑一组对照。这种方式适合单个 Skill 的快速迭代，但不适合我们的场景：我们需要评测的是整个 Agent 工具链的综合能力（包括 Skill 触发、Agent Loop 行为、多工具协作），而且要跨版本、跨模型持续追踪。
因此，我们的核心设计是：将 baseline 固化为一套持久化的评测集（manifest），所有评测维度的基准值预先标注在其中。
评测集以 manifest.json 文件定义，包含 manifest 级元数据和题目数组两层结构：

{

"name": "scene_0328-2",

"created_at": "2026-04-02T20:19:19+08:00",
"created_by": "yue",
"description": "scene_0328-2",
"total_questions": 42,
"questions": [ ... ]
}
manifest 级元数据用于标识评测集的版本、创建者和规模，支持多版本评测集的管理和溯源。

基准值也不是一锤定音的常量——随着模型能力进步和 Skill 迭代优化，当多个版本的评测结果持续超过基准值时，说明原有基准已失去牵引作用，需要上调；反之，场景变化导致基准不合理地严格时也需要下调。我们建议每季度结合最新评测数据进行一轮校准，确保基准始终处于"可达但有挑战"的合理区间。
这套评测集服务于三个目标：

评定任务完成度：Agent 产出的结果是否正确——通过 reference_files 参考答案与 eval_config 规则驱动的自动化比对（如忽略特定列、是否比对样式），结合裁判模型的语义判断
评定执行效率：Agent 在完成任务的过程中，工具调用、Token 消耗、时间、成本是否合理——通过实际值与 baseline_* 基准值的偏差来量化
牵引工具能力提升：每次迭代（新版本 / 新模型 / 新 Skill）都用同一套评测集回归验证，用数据驱动改进方向
评测集的几个关键设计原则：

端到端任务：从原始 Excel/CSV 输入到最终报表（Excel/HTML）产出，不是代码片段而是完整工作流
难度分级：中等 / 困难等多级，覆盖不同任务复杂度
按 Skill 场景组织：同一 skills 下包含多道题目（如 pr-checker 有 9 题、空闲资源分析 有 6 题），覆盖同一 Skill 的不同输入和边界情况
可回归：新版本 / 新模型 / 新 Skill 上线后，用同一题库回归验证，确保能力不退化
4.2 全链路 Trace：评测数据的采集基础
评测集定义了"评什么"——接下来的问题是"用什么数据来评"。
传统软件评测只需看输入和输出，但 Agentic 应用有本质不同：Agent 的行为是涌现的、非确定性的，同一个任务可能走完全不同的推理路径。如果只看最终结果，就无法回答"为什么失败"、"为什么变慢"、"为什么上个版本能做对这个版本不行"这类关键问题。因此，全链路 Trace 是 Agentic 应用评测可解释性和改进闭环的前提条件。
Agent 完成任务时产生的 JSONL 交互轨迹，记录了从会话启动到任务结束的完整执行链路：

{"type":"session_start","model":"Claude-4-Sonnet","user_question":"分析告警日志..."}
{"type":"tool_call","tool_name":"Glob","tool_input":{"pattern":"**/*.xlsx"}}
{"type":"tool_result","tool_result":"found 3 files","tool_error":false}
{"type":"assistant","thinking":"需要先理解数据结构...","text":"我来分析..."}
{"type":"tool_call","tool_name":"Read","tool_input":{"path":"data.xlsx"}}
{"type":"tool_result","tool_result":"[文件内容]","tool_error":false}
{"type":"assistant","thinking":"数据有3列，需要按月份汇总...","text":"数据结构已明确..."}
{"type":"tool_call","tool_name":"Shell","tool_input":{"command":"python analyze.py"}}
{"type":"tool_result","tool_result":"Error: KeyError 'date'","tool_error":true}
{"type":"assistant","thinking":"列名不对，应该是'日期'而非'date'...","text":"修正列名..."}
{"type":"result","status":"success","duration_ms":432220,"input_tokens":488250,"output_tokens":3447,"cost_usd":3.62}
注意上面的 Trace 记录了一次典型的错误-纠正过程（工具调用失败 → 思考修正 → 重试），这类信息在只看最终结果时完全不可见，却对 Skill 优化至关重要。
系统从 JSONL 中自动提取以下过程指标，无需人工介入：
提取指标提取方式用途工具调用总次数、成功/失败次数统计 tool_call + tool_result 记录衡量工具调用可靠性与效率Token 消耗（input/output）从 result 记录提取衡量思考效率思考轮次统计含 thinking 字段的 assistant 记录衡量推理精简度E2E 耗时从 result 记录提取 duration_ms衡量端到端性能API 成本从 result 记录提取 cost_usd衡量成本效率这些指标与评测集中的 baseline_* 基准值做比较，就构成了量化评分的数据基础。同时，完整的 Trace 还服务于更深度的分析——失败根因定位、性能瓶颈识别、版本回归对比、Skill 优化指引等（详见 4.4 裁判模型）。

4.3 评分体系：TTTEC 六维模型 + pass@k 通过率
有了评测集（基准值）和 Trace（实际值），就可以构建评分体系。我们设计了两套互补的指标：TTTEC 六维连续评分量化能力画像，pass@k / pass^k 通过率量化稳定性。

TTTEC 六维模型
将 Agent 的综合能力分解为六个可量化维度：
维度衡量目标核心公式思路数据来源T1 任务完成度做对了吗？eval_config 规则自动化比对 + 裁判模型语义判断参考答案比对 + 评测配置 + 裁判模型T2 工具准确率调用可靠吗？成功次数 / 总次数 × 100Trace 自动提取T3 工具效率调用精简吗？超出 baseline_tool_count 的扣分Trace + 基准对比T4 思考效率Token 消耗合理吗？Token 超标率 + 轮次超标率加权Trace + 基准对比E E2E 性能速度快吗？超出 baseline_time_ms 的扣分Trace + 基准对比C 成本效率花费可控吗？超出 baseline_cost_usd 的扣分Trace + 基准对比其中 T1 依赖参考答案的规则自动化比对与裁判模型的语义判断，T2-C 五个维度完全由 Trace 自动提取的实际值与评测集基准值比较得出。六个维度通过加权求和得出综合分（权重可由用户自定义），支持题目级和版本级两个粒度的分析。
以"工具执行效率"为例：
Stool_eff=max⁡(0, 100−β×max⁡(0, actual_calls−baseline_tool_count))Stool_eff​=max(0, 100−β×max(0, actual_calls−baseline_tool_count))
其中 baseline_tool_count 来自评测集中该题的预设基准值。

通过率指标：pass@k 与 pass^k
TTTEC 六维模型给出的是连续分数（0-100），但在评测运营中还需要一个更直观的二值指标——题目到底"过了没有"。
Agentic 应用的输出具有非确定性：同一道题目、同一个 Skill、同一个模型，多次运行的结果可能不同。单次运行的通过/不通过判定无法反映 Agent 的真实能力水平。因此，我们引入了多次执行（k 次）下的通过率指标体系：
指标定义含义适用场景原始通过率task_completion_score == 1.0 的比例向后兼容的严格通过率（单次完美完成）单次执行场景pass@kk 次中至少 1 次通过的题目比例衡量"能力上限"——Agent 有没有能力做对评估能力覆盖面pass^kk 次中全部通过的题目比例衡量"稳定性"——Agent 能否稳定做对评估生产可靠性通过判定标准：多次执行时，单次执行的通过判定使用 task_completion_score >= 0.6（可配置阈值），而非要求完美的 1.0——这是因为 Agentic 任务的产出往往是复合文件（如 Excel 报表），允许在非关键细节上有偏差。单次执行场景向后兼容，仍使用 == 1.0 的严格判定。
跨题目聚合语义：当同一道题目出现在多个评测任务中时，按题目 ID 折叠聚合——pass@k 取任意一组通过即可，pass^k 要求所有组都满足。
一个具体的例子：

题目 A：执行 3 次 → 通过、通过、失败   → pass@3 = ✓   pass^3 = ✗
题目 B：执行 3 次 → 通过、通过、通过   → pass@3 = ✓   pass^3 = ✓
题目 C：执行 3 次 → 失败、失败、失败   → pass@3 = ✗   pass^3 = ✗
题目 D：执行 1 次 → 通过              → pass@1 = ✓   pass^1 = ✓
版本级汇总（4 题）：
pass@k = 75%（3/4 题至少 1 次通过）
pass^k = 50%（2/4 题全部通过）
pass@k 与 pass^k 的差距是一个关键的质量信号：差距大说明 Agent 能力够但稳定性差（优化方向：让 Skill 指引更确定性）；差距小说明表现稳定（专注于提升 pass@k）；两者都低说明基础能力不足（重新审视 Skill 设计或模型选择）。在版本级 Dashboard、Skill 维度汇总、场景维度汇总中，pass@k 和 pass^k 都与六维得分并列展示，并支持与上一版本的百分点（pp）差值对比。

4.4 裁判模型：基于 Trace 的深度诊断
Baseline 评分解决了"与预设基准比偏差多大"的量化问题，但它本质上是规则驱动的——公式固定、系数预设、不理解语义。面对以下问题，Baseline 无能为力：

Agent 为什么选了一个低效的工具？是不知道有更好的选择，还是 Skill 指引不清晰？
某次工具调用失败后，Agent 的重试策略是否合理？死循环重试 vs 换一种方式？
Agent 是否真正理解了用户意图？还是只在"表面完成"——数据结构对了但语义错了？
Skill 是否被正确触发、读取、遵循？还是 Agent 自由发挥了一套替代方案？
这些需要语义级理解的判断，正是裁判模型（Judge Model）的用武之地。

裁判模型的核心设计
我们的裁判模型评测不是简单地让 LLM"看看结果打个分"，而是将完整的执行轨迹（Trace）作为输入，让裁判模型像一位资深工程师做 Code Review 一样，逐步审查 Agent 的每一个决策和行动。
轨迹预处理：将 JSONL 格式的原始轨迹转化为带编号的结构化文本，每个事件标注 Step N（1-based，与前端 Trace 回放视图保持索引一致），确保裁判模型的分析结论可以精确回溯到具体步骤。

共 25 个步骤。每个步骤的编号 (Step N) 是唯一标识，在你的分析中引用步骤时必须使用该编号。
### Step 1
{"type":"session_start","model":"Claude-4-Sonnet","user_question":"分析告警日志..."}
### Step 2
{"type":"tool_call","tool_name":"Glob","tool_input":{"pattern":"**/*.xlsx"}}
### Step 3
{"type":"tool_result","tool_result":"found 3 files","tool_error":false}
...
裁判模型的四维评分体系
裁判模型独立于 Baseline 进行四个维度的评估，每个维度 0-100 分，并配有分级标准：
裁判维度评估目标打分标尺（摘要）执行效率（Efficiency）动作精简度与试错成本90+：路径极致精简；60-74：明显绕路，3次以上无效操作；<60：死循环工具准确性（Tooling Logic）工具选择合理性与参数精准度90+：精准选用最优工具；60-74：多次参数构造失败；<60：完全选错工具思考效率（Thinking Efficiency）上下文理解与推理效率90+：完美结合上下文；60-74：产生文件幻觉；<60：完全无视报错日志任务完成度（Task Completion）最终结果与用户意图契合度90+：完美解决含边界情况；60-74：表面完成但引入新 Bug；<60：完全偏离意图关键区别在于：裁判模型的打分基于对推理过程的语义理解，而非简单的数值比较。例如，两个 Agent 都调用了 15 次工具（Baseline 工具效率得分相同），但裁判模型会区分"15 次都有明确目的"和"其中 5 次是死循环重试"。

关键步骤分析：问题链溯源
裁判模型不仅打分，还要输出关键步骤分析（Critical Steps Analysis）——只标注有问题的步骤（REDUNDANT 冗余 / ERRONEOUS 错误），正常步骤不列出。每个问题步骤必须包含因果链分析：

{
"step_id": "Step 7",
"assessment": "ERRONEOUS",
"observation": "Agent 尝试用 python analyze.py 执行分析脚本，但使用了英文列名 'date'，实际 Excel 中的列名为中文'日期'",
"context_chain": "前因：Step 5 读取了 Excel 文件内容，返回结果中已明确显示列名为中文'日期'，但 Agent 未仔细解析该返回值。后果：导致 Step 8-10 连续 3 步在修正列名映射上反复试错",
"root_cause": "Agent 未充分解析 Step 5 工具返回的数据结构，直接假设了英文列名",
"expected_action": "应在 Step 5 返回后立即提取实际列名列表，而非凭假设编写脚本"
}
这种因果链分析的价值在于：它不只告诉你"Step 7 错了"，还告诉你"错误的根源在 Step 5 的信息未被利用，导致了 Step 8-10 的连锁浪费"。这直接指向了 Skill 的改进方向——例如在 SKILL.md 中增加"读取文件后必须先确认列名"的指引。

Skill 合规性审计
裁判模型还内置了 Skill 合规性审计机制。评测集中每道题目标注了期望被触发的 Skill（skills 字段），裁判模型会在 Trace 中检查：

是否调用了该 Skill——Trace 中是否存在对应的 tool_call: Skill 记录
是否读取了 Skill 内容——是否有读取 SKILL.md 正文的证据
是否执行了 Skill 要求的脚本/工作流——如果 Skill 中明确要求执行某脚本，Trace 中是否有对应的执行记录
即使 Agent 通过其他路径完成了任务，如果未调用期望的 Skill，裁判模型也会判定为"不合规"并扣分。这确保了 Skill 的触发准确性和遵从度可以被量化追踪。Skill 合规子分按以下规则计算并融入工具准确性维度：
tooling_logic_score=0.3×原工具分+0.7×Skill 合规子分tooling_logic_score=0.3×原工具分+0.7×Skill 合规子分
每个期望 Skill 的合规检查按 10 分制：调用到（+3）、读取说明（+3）、脚本要求满足（+4，若无脚本要求则默认得 4 分），最终线性映射到 0-100。

裁判结果的持久化与复用
裁判模型的评测结果被完整持久化到数据库（JudgeResult 表），包括：

四维分数及其分级、扣分原因
总结结论（conclusion）
关键步骤分析（critical_steps_analysis）
改进建议（evolution_suggestions）
安全风险检查结果
原始响应（raw_response，用于排查解析失败）
持久化的裁判结果服务于多个下游场景：融合评分计算、共性问题分析、前端详情展示、跨版本对比等。

4.5 融合评分：Baseline + 裁判模型的互补
Baseline 评分和裁判模型评分各有优势——前者稳定、可复现、不依赖外部模型；后者语义丰富、能理解过程质量。我们通过 50:50 融合将两者结合：
融合维度Baseline 侧（规则计算）裁判模型侧（语义评估）融合公式任务完成度参考答案自动比对分意图达成度（Task Completion）50% + 50%工具准确性成功次数/总次数 × 100工具决策合理性（Tooling Logic）50% + 50%执行效率超出 baseline 工具次数扣分路径精简度（Efficiency）50% + 50%思考效率Token + 轮次超标率上下文理解力（Thinking Efficiency）50% + 50%E2E 性能超出 baseline 耗时扣分—纯 Baseline成本效率超出 baseline 成本扣分—纯 BaselineE2E 性能和成本效率不参与融合——这两个维度是纯客观的物理量（时间和金额），裁判模型无法提供额外信息。
融合的触发时机是事件驱动的：

JSONL 数据导入或 Pipeline 提交后，系统先自动计算 Baseline 六维分
后台自动触发裁判模型评测（若已启用且配置了 API Key）
裁判评测完成后，自动触发融合评分重算，更新最终的 ScoreResult
系统同时保留纯 Baseline 分（baseline_* 字段）和融合后分（顶层字段），前端可以分别展示两套评分体系的结果，便于用户理解裁判模型带来的评分变化。

4.6 版本级共性问题抽取
单题的裁判评测回答的是"这道题 Agent 做得怎么样"，但更有价值的问题是：这个版本的 Agent 整体存在哪些系统性问题？ 靠人工逐题翻看裁判报告来归纳共性问题既低效又容易遗漏。
为此，我们实现了共性问题自动抽取——对裁判模型进行二次调用，将版本内所有单题裁判结果作为输入，由 LLM 提炼出跨题目的共性模式。

共性分析的输入构建
共性分析并非将原始 Trace 再喂一次给模型（那样的上下文开销不可承受），而是汇总每道题的裁判评测摘要：

[
{

`"question_id": "project_business_analysis-0001",`

`"question_name": "5G核心网项目经营分析1",`

`"category": "Excel",`

`"overall_score": 72,`

`"conclusion": "Agent整体完成了任务，但在Step 3-5存在冗余目录遍历...",`

`"deductions": {`

  `"execution_efficiency": {"score": 65, "level": "及格", "reason": "Step 7-9连续3次因正则错误导致grep失败"},`

  `"tool_accuracy": {"score": 80, "level": "良好", "reason": "Step 12使用了bash命令替代专用工具"},`

  `"thinking_efficiency": {"score": 70, "level": "及格", "reason": "Step 5的报错信息未被利用"},`

  `"task_completion": {"score": 75, "level": "良好", "reason": "核心功能完成但缺少边界检查"}`

`},`

`"evolution_suggestions": [`

  `{"type": "execution_path", "suggestion": "应使用grep -r全局搜索替代逐层目录遍历"},`

  `{"type": "system_prompt", "suggestion": "建议增加「读取报错后先分析原因」的约束"}`

`]`

},
...
]
这种"摘要级"输入使得一次共性分析可以覆盖一个版本内的所有题目（数十道），而不会超出上下文窗口限制。

共性分析的输出结构
共性分析输出三个层次的信息：
1. 整体概述（summary）——200 字以内的版本能力画像，包含主要优势和关键不足。
2. 共性问题（common_issues）——3-6 条跨题目高频出现的问题模式：

{
"dimension": "执行效率",
"issue": "Agent 在遇到工具调用失败时频繁采用死循环重试策略，而非分析报错信息后切换方案",
"frequency": "约 60% 的题目存在该问题",
"impact": "平均每题多消耗 3-5 个无效步骤，导致执行效率维度均分仅 65",
"question_ids": ["project_business_analysis-0001", "alarm_analysis-0003", ...],
"examples": ["5G核心网项目经营分析1中Step 7-9的重试循环", "告警分析3中Step 12-15的路径猜测"]
}
每条共性问题的 question_ids 字段经过白名单校验——只允许输入数据中真实存在的题目 ID，防止模型编造不存在的引用。
3. 改进建议（improvement_suggestions）——3-5 条可落地的系统级优化方向：

{
"type": "error_handling",
"priority": "高",
"suggestion": "在 Skill 中增加错误恢复策略指引：遇到工具调用失败时，要求 Agent 先读取报错信息、识别错误类型（参数错误/权限不足/资源不存在），再根据类型选择不同的恢复路径，而非直接重试",
"expected_impact": "预计可将工具失败后的平均恢复步骤从 4.2 步降低到 1.5 步"
}

4.7 多版本横向对比与趋势追踪
三层评测体系解决了"怎么评"的问题，接下来的关键是"怎么比"——评测集的另一个核心价值是支持多版本横向对比。这里的"版本"是一个宽泛的概念——它可以是同一模型的 Skill 迭代版本、不同 LLM 模型的对比等，甚至是同一配置的多次回归验证。本质上，任何需要对比的两组评测结果，都通过不同的版本号来组织。
主要包括：

雷达图：6 维得分叠加对比，一眼看出版本间的能力差异分布
热力图：题目分类 × 维度 → 得分矩阵，定位薄弱环节
趋势折线图：版本序列的得分变化趋势
通过率对比：pass@k / pass^k 的版本间百分点差值（pp），追踪稳定性变化
差异标注：自动计算 ΔS 和 Δ%，升降箭头 + 颜色编码
在 Skill 维度和场景维度的分组汇总中，每个分组同时展示 pass@k / pass^k 通过率和六维均分，便于定位哪个 Skill 或场景的稳定性最薄弱。
这个设计背后的产品假设是：评测的核心价值不在于一次性判断"好不好"，而在于持续追踪"是否在变好"——每次版本迭代后，通过多版本对比观察总分趋势和 pass@k 与 pass^k 的差距变化，确认能力在提升、稳定性在收敛。

五、经验总结与关键认知
5.1 方法论层面
基于官方方法论，我们结合产品的实际情况补充完善了相关的认知：

Skill 评测必须是对照实验：没有 baseline 就没有评价标准。你以为 Skill 有效，可能只是模型本身就能做到。
基准值是"活的"：评测基准不是一锤定音的常量。模型能力在进步、Skill 在迭代、业务场景在变化——我们建议每季度根据最新数据重新校准。
断言本身需要评测：这是一个元问题。always-pass 的断言给人虚假安全感；always-fail 的断言浪费时间。skill-creator 的 Grader 已经有"eval_feedback"机制，我们在此基础上做了自动化检测。
Skill 的"伤害"也要量化：Skill 并非总是正面的。我们观察到三种负面效应：
Skill 过长导致上下文窗口浪费
过度约束导致 Agent 在非标准输入面前僵化
Skill 指令与模型内在行为冲突导致"内耗"
因此 baseline 对照组永远不能省略——它既是 Skill 效果的参照，也是 Skill 负面影响的检测器。
规则评分和语义评分缺一不可：Baseline 规则评分稳定、可复现、成本低，但无法理解过程语义；裁判模型能做深度诊断，但依赖外部 LLM、有成本和延迟开销。50:50 融合是我们经过多轮实验验证的平衡点——偏重任何一方都会导致评分偏差（纯 Baseline 忽略过程质量，纯裁判波动较大且成本高）。
共性分析是改进的加速器：逐题看裁判报告太慢，靠人脑归纳容易遗漏。让 LLM 对所有裁判摘要做二次分析后，我们发现了多个靠人工难以察觉的系统性问题——例如"70% 的题目中 Agent 遇到工具失败后不读报错信息直接重试"。这类洞察直接转化为 Skill 中的错误恢复指引，带来了批量性的改进效果。
评测集设计是评测质量的上限：规则评分和裁判模型都依赖于评测集的质量——reference_files 的标准答案是否准确、eval_config 的比对规则是否完备、baseline_* 基准值是否合理，直接决定了自动化评分的可信度。Agent 可能在所有量化指标上表现优秀，但如果评测集本身的标准答案有瑕疵或比对规则不够精细，评分就会产生误导。因此，评测集的持续校准和迭代与 Skill 的迭代同等重要。
